# 面试准备笔记

> 基于「AI 知识库助手」项目，针对 AI Agent 应用开发初级岗。

---

## 一、项目介绍（2-3 分钟版本）

我做的项目是一个 RAG 知识库问答系统，解决的问题是：当你有一堆文档想快速查找某个知识点时，不用一个个搜索，直接问就行。

技术栈是 Python + LangChain LCEL + DeepSeek，Embedding 用了 bge-small-zh-v1.5，向量库是 ChromaDB，前端 Streamlit，部署用 natapp 做了内网穿透，面试时可以直接点链接试用。

**核心链路：** 用户上传文档 → 切片 → Embedding 向量化 → 存入 ChromaDB。提问时，问题先经过向量检索找到最相关的 3 个片段，拼进 Prompt，再由 LLM 生成回答。这条链路用 LCEL 管道语法实现——检索上下文、对话历史、原始问题三个输入分支并行进入 Prompt，LCEL 的 lazy evaluation 保证只有变化的分支才会重新计算。

**智能路由的设计决策：** 到底是先判断问题类型再决定走不走 RAG（前置路由），还是先走 RAG 再看结果？我选了后者。因为 LLM 说"不知道"的成本很低，但如果先分类分错了，好的检索结果就永远用不上了。我的逻辑是：RAG 先检索，LLM 自己判断能不能用检索结果回答，不能的话降级到自身知识，并且降级时明确标注"知识库中未找到"，杜绝编造来源。

**Embedding 单例与 Loader 非常例的判断：** Embedding 模型加载一次要 150MB+ 内存，整个应用生命周期只需加载一次，所以用单例。但文档 Loader 每次上传都要新建实例，因为文档内容变了。面试官问你"什么该单例、什么不该单例"，这就是答案。

**部署：** 用 natapp 做内网穿透，本地跑服务直接暴露公网链接。HuggingFace Spaces 和 Render 都试过——前者国内访问不稳定，后者免费计划太慢——最终选了最轻量的方案。

## 二、高频追问及回答要点

### RAG 核心链路

**Q: 切片大小设的多少？为什么？**

500 字符 + 50 重叠。太大信息不完整，太小检索不准，500 是实验后的经验值。中文 1 字符 ≈ 1-2 token，500 字符大约 500-700 token，远在 DeepSeek 64K 窗口内。

**Q: 检索出 3 个片段，但如果它们说的是同一件事怎么办？**

引出了 MMR 策略：不只取最相似的，还考虑多样性。`fetch_k=12` 先取 12 个候选，`lambda_mult=0.7` 做多样性重排，最终返回不重复的 3 个。

**Q: 用户问的问题跟文档里的文字不完全一样，怎么搜到？**

靠 Embedding 语义检索，不是关键词匹配。比如文档写"MySQL 的事务特性"，用户问"数据库的 ACID"，语义上相近就能命中。

**Q: 切 chunk 之后向量库要重建吗？**

必须重建。chunk 变了 → 向量变了 → 旧向量不匹配。

---

### LLM 与 Prompt

**Q: Prompt 为什么那样设计？**

① 先给身份角色约束风格；② 对话历史在前、上下文在后——让 LLM 先理解对话背景再看文档；③ 明确"没找到就说不知道"控制幻觉；④ `<参考文档>` 标签包裹防止 prompt injection。每一条都有明确的理由。

**Q: 超长文档超过 LLM 上下文窗口怎么办？**

只检索 top-3 片段，每个 500 字，总共约 1500 字，远在 DeepSeek 64K 窗口内。切片 + 检索本身就是应对长文档的方案。

---

### 向量库与 Embedding

**Q: 为什么选 ChromaDB 而不是 FAISS？**

ChromaDB 自带持久化，不需要额外序列化。FAISS 检索更快但关掉就没了。你的场景是小规模、要持久化，ChromaDB 刚好。

**Q: 为什么选 bge-small-zh-v1.5？**

① 中文场景优化；② small 级别内存占用小（CPU 可跑）；③ 社区验证过的中文 Embedding Leaderboard 前列。

---

### 工程能力

**Q: 这个系统如果每天有 100 个用户用，哪里最先崩？**

① ChromaDB 单机版本，并发写会锁；② Embedding 在 CPU 上跑，高并发下慢。知道瓶颈在哪比知道解决方案更重要。

**Q: .env 能推上 GitHub 吗？**

不能。`.gitignore` 里已排除。面试官确认你有安全意识。

---

### 陷阱题

**Q: RAG 一定有幻觉吗？怎么控制？**

幻觉不可能完全消除，但可以多种策略组合控制：① Prompt 约束"不知道就说不知道"；② Agent 路由标注来源让用户自行判断；③ 检索片段不相关时让 LLM 诚实降级而不是硬编。

**Q: 你做的跟直接用 ChatGPT 传文件问答有什么区别？**

① 数据在本地，不经第三方；② 检索策略、切片大小完全可控；③ DeepSeek 成本远低于 ChatGPT；④ 证明了你会搭 RAG pipeline——去公司就能给它们搭自己的。

## 三、面试区分度速记

面试时主动提到的三个点，能让你从"也做过 RAG"变成"理解 RAG"：

1. **LCEL 的 lazy evaluation** — 只有变化的分支重新计算，不变的分支复用缓存
2. **Embedding 单例 vs Loader 非常例** — 大模型只加载一次，Loader 每次新建
3. **前置路由 vs 结果反推** — 选"先检索再判断"的理由：分类错误成本 > 降级成本
