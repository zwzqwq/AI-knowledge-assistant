## 2026-07-09: 架构讨论 — RAG 系统中"文档优先"vs"知识优先"的设计取舍

- **类型**: 架构设计 / 设计哲学
- **严重程度**: 设计决策（影响系统行为边界）
- **问题**: 在 RAG 知识库项目中，LLM 应该先查文档再回答，还是先用自己的知识回答再用文档补细节？有没有行业标准？
- **分析**: 业内没有统一规定，但有两派设计哲学：

**派系 A：文档优先（Retrieval-First）**
```
用户问 → 检索文档 → 用文档内容回答 → 文档没有才降级到自身知识
```
- 假设前提：知识库是权威信息源，文档内容比 LLM 训练数据更新、更准
- 适用场景：法务合规系统、客服 FAQ、医疗知识库——答案必须和官方口径一致
- 优势：幻觉可控，LLM 被文档"锚定"住
- 劣势：文档质量差时答案也差；检索不到就回答"不知道"

**派系 B：知识优先（Knowledge-First）**
```
用户问 → 先用自身知识给核心答案 → 再查文档补企业特有的细节
```
- 假设前提：LLM 基础知识可靠，知识库的价值是补充 LLM 不知道的企业特定信息
- 适用场景：企业 Wiki 问答、技术文档助手——常识性问题 LLM 可以回答，文档提供具体配置/规范
- 优势：用户体验好，不会连"MySQL 是什么"都依赖文档
- 劣势：LLM 可能过度自信，用自己的知识覆盖文档中修正性的信息

- **本项目的选择**: 从文档优先起步，发现两个问题后切换到知识优先——①常识性问题（"MySQL 是什么"）被文档粒度限制导致回答不完整（只输出引擎侧边栏），②过度约束让 LLM 回避（Answer Collapse）。当前文档是用户上传的技术文章而非企业规范，知识优先更合理

- **进一步的企业级方案**: 不是二选一，而是分层路由——在 router 层做**问题分类**，常识类走知识优先，企业特定类强制文档优先。真正的架构设计不是在 prompt 里修补，而是在路由层做策略分发

- **面试怎么讲**: "RAG 系统的回答策略没有万能解。我的设计演进是：第一阶段文档优先→发现常识问题回答不完整→第二阶段切换到知识优先+文档补充→回答质量明显提升。如果要上生产，我会在 router 层做问题类型分类，常识类走混合策略、企业规范类强制文档锚定。这是架构层面的策略分发，不是 prompt 层面的补丁。"

## 2026-07-08: 检索不相关——v3 终极版本：两层检索均无完美结果

- **类型**: 检索质量 / 系统瓶颈
- **严重程度**: 间歇性行为异常
- **现象**: 用户问"Redis AOF与RDB"，知识库无直接内容（只有哨兵文档）。router 正确触发 web_search，但 Bing 返回的也是简介/安装教程/官网首页。LLM 最终靠自身知识回答——正确行为。但来源标签可能错误标为 knowledge_base/web_search
- **根因**: 不是 prompt 的 bug——是 RAG 系统的天然瓶颈。本地知识库 + 搜索引擎都无法覆盖所有技术细节。LLM 在两种不相关内容之间来回搜索，最终只能用自身知识回答
- **现状**: generate prompt 已通过严格的主题匹配+来源文件名检查，能正确识别 80% 的不相关案例。剩余 20% 是搜索质量的限制

## 2026-07-08: 检索结果"答非所问"再次出现——v2 更隐蔽的版本

- **类型**: 幻觉 / Prompt 设计
- **严重程度**: 行为异常
- **现象**: 用户问"详解Redis的AOF与RDB"，知识库中实际有 redis 哨兵和集群文档。ChromaDB 返回了 redis 相关内容（主从复制、哨兵等），但**不包含 AOF 和 RDB 的直接解释**。LLM 看到 "Redis" 关键词就生了套——从 Redis 主从复制的搜索结果中生成了"生产环境必须开启RDB或AOF"的回答，答非所问
- **根因**: 检索到的文档虽然是 Redis 主题，但**只讨论哨兵和集群架构**，没有深入 AOF 与 RDB 的细节。LLM 看到同主题的关键词后就"忠实"地从可用信息中拼凑回答——这是 v2 更隐蔽的幻觉版本：主题看似匹配、内容实则偏题
- **修复（三层防御）**:
  1. **检索片段带上 source（文件名）**：`retrieve_node` 的片段标注来自哪个文件，让 `generate_node` 的 LLM 看到结构化的来源信息
  2. **Generate prompt 重写为严格的过濅规则**：说清"如果所有结果都是关于技术A但用户问技术B的子细节→不可用"，强调"不要因为有相同关键词就生搬硬套"
  3. **Router prompt 追加相关性判断**：LLM 看完检索结果后必须判断→若不直接相关→调 web_search 补充博客/官方文档
- **面试怎么讲**: "RAG 的准确性不取决于检索的召回率，更关键的是如何过滤不相关结果。我从两次 bug 中总结出三层防御——结构化源信息→严格过濅 prompt→路由兜底 web_search。"

## 2026-07-08: 知识图谱重复抽取——每次上传都全量重新抽取实体关系

- **类型**: 性能 / 去重缺失
- **严重程度**: 行为异常（非阻断，但浪费 Tokens）
- **现象**: 用户每次上传文档时，`_build_knowledge_graph()` 都对所有 chunk 重新调用 LLM 抽取实体关系。同一文档多次上传、已有图谱的情况下，重复抽取是多余的——实体关系已经存在于图中
- **根因**: `add_document()` 直接无条件调用 `_build_knowledge_graph()` → `extract_from_chunks()` 遍历所有 chunk 逐个调 LLM 抽取——没有检查"这个文档是否已经抽取过了"
- **修复**: 
  1. `ChatService` 增加 `_kg_doc_names: set` 记录已抽取过的文档名
  2. `add_document()` 只在 `filename not in self._kg_doc_names` 时才调用图谱构建
  3. `_build_knowledge_graph()` 增加 `filename` 参数，抽取完成后 `self._kg_doc_names.add(filename)`
- **面试怎么讲**: "批量操作的去重是第一道防线。知识图谱构建是昂贵操作（每次调 LLM 需要几百 ms），不合适的去重会让服务成本和延迟翻几倍。我在文档入库时增加了一个简单的 filename 去重集合，避免对同一文档重复抽取实体关系。"

## 2026-07-08: 追问"第3点"时 LLM 答非所问——回答被截断导致信息丢失

- **类型**: 上下文截断 / 信息丢失
- **严重程度**: 行为异常
- **现象**: 用户上一轮收到包含 5 个编号关键点的长回答，追问"详解第三点"，LLM 回答了一个完全不相关的内容（把"第三点"理解成了无锁方案的适用场景，而不是原回答中真正的第3条内容）。用户追问的内容本身是明确的，但 LLM 完全理解错了
- **根因**: `ChatService` 将上一轮回答截断到了 500 字符（`[:500]`），恰好截在了原回答的"1. CAS"、"2. CAS的三个大坑"之后、真正的"3. 常见的无锁数据结构"之前丢失了。LLM 看到的只是 1 和 2 的残篇，不知道第 3 点是什么，只好根据"无锁方案"这个主题**猜**第 3 点是什么——猜成了"适用场景/优缺点分析"，而不是原始回答中的"无锁数据结构"
- **核心理解**: 用户追问"第 N 点"时，N 的索引依赖于**完整的上轮回答才能定位**。截断 `[:500]` → 第 3 点在 501 字符后 → LLM 永远看不到 → 永远答不对
- **修复**: 取消 `[:500]` 截断，改为把完整的 `last_answer` 全文传给 LLM。同时去掉"不要调用工具检索"的硬约束（追问概括性话题时可能需要检索）
- **面试怎么讲**: "对话历史的截断策略需要根据追问类型动态调整。'第3点'这种编号式追问需要完整的上文才能定位索引；'继续'这类续写式追问可能需要最后一段即可。我的教训是：一刀切截断 ≈ 丢失关键信息 ≈ 对不齐用户的语义锚点。"

## 2026-07-08: 追问场景下 Agent 仍调用工具——"替代方案"被当成新检索词

- **类型**: 工具滥用 / 决策偏差
- **严重程度**: 行为异常  
- **现象**: 第一轮用户问"举出一个多线程问题并给出解决方案"，LLM 用自己的知识回答（给出了 SimpleDateFormat 的方案，用到 DateTimeFormatter/ThreadLocal 等替代途径）。用户追问"详细讲讲你这个替代方案"——LLM 却发起 retrieval 搜索"替代方案"这个词，然后在 MySQL 文档里搜到 `order by` 索引相关内容，把 `order by` 当成"替代方案"回答了。用户看到的完全牛头不对马嘴
- **根因**: 一个连锁反应：
  1. `ChatService` 把对话历史拼入 `HumanMessage` 发给 router
  2. LLM 看到历史里自己回答过长（上千字），其中混杂了 `替代方案` 多个关键字
  3. LLM 认为 `替代方案` 是用户关心的核心词（而非 "DateTimeFormatter 的替代方案"），决定工具检索
  4. ChromaDB 返回 MySQL 文档中关于 `order by` 的片段——和用户本意完全无关
  5. `generate_node` 的 prompt 虽然更新过（要求判断相关性），但面对"有内容就输出"的默认倾向，还是拼出了回答
- **修复方案**:  
  1. 对话历史截断到 300 字符——不要让长回答淹没用户意图  
  2. 改写 context_message 为 "以下是你和用户的对话历史，用户现在在追问" + 用户当前消息用 `「」` 包裹——结构清晰，LLM 更容易识别追问  
  3. router 检测到 `对话历史` 关键词时，追加 `⚠️ 重要：严禁调用任何工具`——强硬阻止不必要的检索
- **核心教训**: Agent 的决策质量不取决于 router prompt 多聪明，而取决于**传入消息的质量**。消息中**长回答 > 短追问**时，LLM 被长回答关键字带偏——这就是"信号被噪声淹没"问题的根因
- **面试怎么讲**: "多轮对话中，保证信息传递质量比 Agent 算法本身更重要。我从一个很实际的 bug 中学到的：当对话历史过长时，LLM 会被上面回答里的关键词带偏，发起完全不相关的工具调用。解决方式是——①压缩历史长度，②用 `「」` 包裹当前消息做显式区隔，③追问模式下禁止工具调用。"

## 2026-07-10: `context` 字段——AgentState 中写入但从不读取的死字段

- **类型**: 状态管理 / 数据一致性
- **严重程度**: 低（冗余但暂不影响功能）
- **背景**: AgentState 中有 `context` 字段（string 类型），三个工具节点都返回 `"context": context`。同时每个工具调用的 result 也通过 `tool_calls[i]["result"]` 存储。同一个检索结果被存在两处
- **数据流分析**:
  ```
  写入路径（两条）:
    1. tool_calls[i]["result"] = "检索结果..."  ← 结构化，与 name/args/id 关联
    2. context = "检索结果..."                    ← 覆盖式，不知道来自哪个工具
  
  读取路径:
    generate_node: 遍历 tool_calls → 取 tc["result"]  ← 不用 context
    chat_service:  遍历 tool_calls → 取 tc["result"]  ← 不用 context
  ```
  **结论**: `context` 字段是写入但从不读取的死字段。工具结果已通过 `tool_calls` 列表完整传递
- **P7 级判断**: 保留它有隐患——①语义误导：新加入的开发者看到 `context` 字段会以为它是"当前上下文"；②覆盖风险：多工具场景下最后一个工具的 result 覆盖前一个。但由于删掉它涉及 4 个文件的改动（state.py + graph.py ×3 处 return + chat_service.py initial_state），属于跨文件重构。**当前不改，记录在案**
- **面试怎么讲**: 不需要主动提。如果被问到 AgentState 设计："字段遵循单一数据来源原则——工具结果统一通过 tool_calls 列表管理，每个调用带 name/args/result/id 四元组。context 是最早期单工具时代的遗留字段，在多工具架构下已被 tool_calls 替代。"

## 2026-07-10: Router 代码兜底讨论——prompt 约束 vs 代码兜底的架构选择

- **类型**: 架构设计讨论 / 容错策略
- **严重程度**: 设计决策（暂不改动）
- **背景**: 用户问"redis 怎么和 springboot 项目连接"，知识库只有 Redis 哨兵文档，retrieve 正常返回了哨兵内容（不相关），router 没有调 web_search 而是反问用户。经过 prompt 重写（角色→"路由器"、temperature=0），这个问题大幅减少但偶发仍存在。用户提出一个思路：在代码层兜底——router 试图直接回答时，代码强制插入 web_search 调用
- **讨论**: 这个思路的优势是"确定性兜底"——代码不会忽略指令。但代价包括：①代码兜底绕过了 LLM 的决策权，可能在不合适的场景也强制搜索；②兜底逻辑会让 router 代码更复杂，而 router 本应保持简洁；③更好的长期方案可能是进一步的 prompt 优化（去掉 router"直接回答"能力，让 generate 专门负责）
- **当前决策**: 暂不加代码兜底。先测试当前 prompt 改动（角色定义+temperature=0）的实际效果——如果 20 次测试中不相关检索的反问率低于 10%，prompt 方案就够用了。代码兜底是下一条防线，不需要提前部署
- **P7 级设计原则**: "纵深防御是对的，但防线之间要有比例。prompt 层花费 20% 的复杂度覆盖 90% 的场景，代码兜底花费 80% 的复杂度覆盖剩余 10%。在 prompt 还有优化空间时，加代码防线收益递减。正确的顺序是先把 prompt 调到天花板，再考虑代码兜底。"

- **类型**: 状态管理 / LangGraph 机制
- **严重程度**: 阻断（工具结果被丢弃，LLM 基于空信息输出垃圾回答）
- **现象**: 启动后第一次对话随机出现——router 正确调了 retrieve+graph_query，retrieve 正常执行，但第 3 次 router 调用时 LLM 突然输出"你好！我是知识库助手。请问有什么可以帮你的？😊"。明明是技术问题，查询也正确执行了，但回答完全无关
- **日志分析**: 从完整日志追踪 `tool_calls` 的生命周期发现：
  ```
  第 1 次 router 返回: tool_calls = [retrieve(r=""), graph_query_A(r="")]
  retrieve_node 执行 → retrieve.result 被填充 ✓
  第 2 次 router 返回: tool_calls = [graph_query_B(r=""), web_search(r="")]
                        ↑↑↑ 完全新建的列表！第 1 次 retrieve 的 3000 字结果被丢弃！
  第 3 次 router 收到:  tool_calls = [graph_query_B(r=有值), web_search(r="")]
                        ↑↑↑ 只剩 graph_query 的结果，且和 Java 连接完全无关
  → LLM 看到只有一个无关的实体关系 → 不知道该回答什么 → "你好！请问有什么可以帮你的？"
  ```
- **根因**: LangGraph 的 TypedDict State 默认使用"最后写入者胜"规则。`router_node` 第 155 行 `return {"tool_calls": tool_calls}`——每次 router 调用都返回**全新构建**的 `tool_calls` 列表，旧列表（包含已执行工具的结果）被无条件覆盖。V2 因为"深表歉意"那个改动让 router 只调一次工具就结束，掩盖了这个 bug。改回来后又暴露了
- **修复**: 在 `router_node` return 前，将新工具调用**合并**到已有 tool_calls 列表中，而非替换。已执行过且有结果的同名工具不会被重复添加。核心代码：
  ```python
  merged_calls = state.get("tool_calls", []).copy()
  for new_tc in tool_calls:
      already_done = any(old["name"] == new_tc["name"] and old["args"] == new_tc["args"] and old["result"] for old in merged_calls)
      if not already_done:
          merged_calls.append(new_tc)
  return {"tool_calls": merged_calls, ...}
  ```
- **面试怎么讲**: "LangGraph 的 TypedDict State 默认是覆盖语义，不是追加语义。Router 节点每次决策产生一组新的 tool_calls，如果我直接 return 新列表，旧列表就会被 LangGraph 的 reducer 覆盖——包括已经执行完成的检索结果。这个 bug 暴露了我对 LangGraph 状态管理机制的盲区——不是每个循环都重新开始，而是每个节点都在修改同一个 state 对象。解决方案是在 router 返回前合并新旧 tool_calls，保留已执行的结果。"

- **类型**: Prompt 设计 / 系统性迭代
- **严重程度**: 行为异常 → 逐步修复 → 最终稳定
- **背景**: generate_node 的 prompt 经历了 5 轮迭代才稳定。这不是"改一个词就修好"的简单 bug，而是一个系统性的 prompt engineering 学习过程。每一步暴露了一个不同的失效模式。

**第 1 轮：朴素方案 → Answer Collapse**
- 做法: 用大量负面约束控制幻觉——"你必须严格按以下规则""先检查来源文件名""如果文档讲哨兵但用户问 AOF=不相关""禁止硬凑答案"
- 现象: 用户问"MySQL 和 Redis 有什么区别"，LLM 回复"你想了解 MySQL 的哪方面内容？"——完全回避问题
- 根因: LLM 对负面表述的应激反应是收缩行为空间。面对灰色地带，不确定能不能做 → 选最安全的路（反问）。"禁止"是无限集合，"允许"是有限集合
- 修正方向: 正向引导替代负面约束

**第 2 轮：正向引导 → 规则盲从**
- 做法: "检索结果中和用户方向一致的文档就用它来回答""完全不相关才说没找到"
- 现象: 用户问"什么是 MySQL"，文档有引擎对比。LLM 回答"MySQL 有两个引擎：InnoDB 和 MyISAM。你想了解吗？"——用了文档，但文档粒度太细，答不了概括性问题
- 根因: 规则本身有逻辑缺口——覆盖了"方向一致"和"完全不同"，漏掉了"部分相关但粒度不匹配"的灰色地带。LLM 照规则办事，不自己补路
- 修正方向: 把 prompt 从"规则清单"改为"思考路径"

**第 3 轮：知识优先 → 混合不稳定**
- 做法: "先用自身知识回答核心问题，再检查检索结果补充细节"
- 现象: 一半回答是正确的（知识部分），一半是反问/列选项（文档部分太薄就反问填充）
- 根因: LLM 在"知识优先"和"文档优先"两种模式间摇摆。文档内容丰富时表现好，文档稀疏时就退回反问模式
- 修正方向: 需要更强的结构约束，告诉 LLM"回答必须长什么样"

**第 4 轮：长度+结构约束 → 回答偏短**
- 做法: "严格禁止反问""先给完整定义""回答要详细、结构化，用标题和列表组织内容，像写技术文档一样""至少 300 字"；同时新增 `config.LLM_MAX_TOKENS=2048` 配置化控制输出长度
- 现象: 回答不再反问，结构清晰，但内容偏短偏浅——约 200-300 字，类似维基百科摘要而非技术百科条目
- 根因: 规则告诉 LLM"要详细"但没给具体榜样。LLM 对"详细"的理解 = 比你之前写的"简洁"多 100 字。同时也验证了回答"简陋"的真正原因——之前的 prompt 里"友好简洁"四个字直接限制了输出长度
- 修正方向: 规则不够，需要示例

**第 5 轮：示例驱动 → 稳定 ✓**
- 做法: 在规则前加了一个完整的 `<example_input>/<example_output>` 示例——展示了"MySQL 是什么"的期望回答结构（一个大段落定义 + 一个补充段落来自检索）
- 最终 prompt 结构: `示例 → 规则 1（禁止反问） → 规则 2（知识主体） → 规则 3（文档补充） → 规则 4（长度要求）`
- 最终效果: 三个测试问题（"什么是 Redis""详细介绍一下 Redis 各种数据类型""Hash 底层实现"）全部通过——知识主体充实、文档补充自然、不反问、深度匹配问题
- 核心发现: **示例比规则更有效。** LLM 对自然语言规则的理解有偏差（"详细"=多长？），但对示例的模仿非常精准。给 LLM 一个明确的输出样例，比写 10 条规则都管用

- **面试怎么讲**: "我花了 5 轮迭代才把 generate prompt 调到稳定。这个过程教给我三件事：①负面约束导致 Answer Collapse——告诉 LLM 做什么比告诉它不做什么效果好 10 倍；②规则有逻辑缺口 LLM 不会自己补——prompt 不是规则清单，是思考路径；③最重要的一条——示例比规则更有效。LLM 对模糊的自然语言约束理解偏差很大，但对具体样例的模仿极其精准。最终方案是'示例引导 + 4 条规则 + 配置化 max_tokens'。面试官听到这个迭代过程，会比听到'我调了几次 prompt'印象深刻得多。"

## 2026-07-09: `astream()` 返回值结构与 `invoke()` 完全不同

- **类型**: API 理解 / 框架机制
- **严重程度**: 阻断（Agent 生成了答案但 chat_service 全部丢弃）
- **现象**: 把 `chat_service.stream_chat()` 从 `agent.invoke()` 改为 `agent.astream()` 后，三个测试问题（Redis 数据类型、MySQL vs MongoDB、MySQL 存储结构）的回答全部错误——来源标签显示"AI 自身知识"，回答内容为"（Agent 未生成回答）"。但 Agent 本身运行正常（日志显示 router→retrieve→generate 全部正确执行，generate 节点生成了 1000+ 字符的答案）
- **根因**: `invoke()` 返回完整 state（`{"messages": [...], "tool_calls": [...], "final_answer": "..."，...}`——平铺的 dict），`astream()` 返回的**不是同一个东西**——每次 yield 的是 `{节点名: {该节点更新的字段}}`（嵌套的 dict，如 `{"generate": {"final_answer": "..."， "messages": [...]}}`）。代码写了 `final_state.get("final_answer"， "")` 去平铺取，永远拿不到（因为 `final_answer` 被包在 `final_state["generate"]` 里面）。Agent 每次生成完整答案，chat_service 拿不到直接丢弃，然后输出"Agent 未生成回答"
- **修复**: 遍历 `astream` 的每个 chunk，按节点名分发处理——`chunk["router"]` 取 router 的 final_answer、`chunk["retrieve"]`/`chunk["graph_query"]` 收集 tool_calls、`chunk["generate"]` 取最终答案。数据结构变了，代码也必须跟着变
- **面试怎么讲**: "LangGraph 三种调用方式的返回结构各不相同：`invoke` 返回完整平铺 state，`astream` 返回逐节点增量 `{节点名: {字段}}`，`astream_events` 返回事件粒度。选哪个取决于你的场景——批处理用 invoke，进度展示用 astream，真正的逐字流式用 astream_events。我踩的坑是没注意到 astream 的嵌套结构，用 invoke 的取值方式去解析它。这是框架迁移中最容易犯的错误——API 名字相似，数据契约完全不同。"

## 2026-07-06: 多轮对话"失忆"——Agent 上下文丢失

- **类型**: 上下文丢失 / 设计缺陷
- **严重程度**: 行为异常
- **现象**: 前一轮在聊 MySQL 知识，下一轮只发了"索引"两个字，Agent 回复"我没有获取到任何外部信息，请问有什么我可以帮你的？"——好像完全不记得刚聊过什么
- **根因**: `ChatService.stream_chat()` 传给 Agent 的 `messages` 只有当前这一条消息，没有注入上一轮对话历史。Agent 拿到孤零零的"索引"这个词，既不像查询也不像问题，LLM 就没触发任何工具调用
- **修复**: 在 `initial_state["messages"]` 前拼上 `ConversationHistory.format()` 的格式化文本：`"对话历史：\n{history}\n\n用户当前问题：{message}"`
- **面试怎么讲**: "多轮对话的上下文管理是 Agent 系统的关键工程挑战。LangGraph 的 StateGraph 给了你状态容器，但填什么内容进去、怎么填，需要自己设计。我踩过的坑就是——graph 本身不保存历史，每次 invoke 要手动注入。"

## 2026-07-06: 检索结果"答非所问"——向量相似但语义不相关

- **类型**: 幻觉 / Prompt 设计
- **严重程度**: 行为异常
- **现象**: 用户问"Spring Boot 如何搭建"，知识库里只有 MySQL 文档。Agent 走了 RAG 检索，ChromaDB 返回了 MySQL 片段（向量相似度匹配到的），`generate_node` 忠实地把 MySQL 存储引擎、连接池等作为答案输出——用户一脸问号
- **根因**: `generate_node` 的 system prompt 太软——"如果知识库有相关内容，优先使用知识库内容"。LLM 看到的"内容"就是 MySQL 片段，它就老实照搬了。**Agent 的检索节点不判断相关性**，检索和生成是两个独立的步骤
- **修复**: 在 prompt 中加入相关性判断指令：`"如果搜索结果和用户问题不相关（例如用户问Spring Boot但搜到了MySQL），诚实地说'知识库中未找到关于XX的内容'"`——让 LLM 在生成时做最后一道相关性过滤
- **面试怎么讲**: "向量检索的本质是语义相似度，不是语义相关性。'Spring Boot'和'MySQL连接池'在同一个文档里，向量空间中会被拉近。我的解决方案是在 Generate 节点的 prompt 里加相关性判断指令——让 LLM 判断检索结果是否真的能回答用户问题，不能就诚实降级、用自己的知识补。这是成本最低的幻觉控制，属于 Prompt Engineering 层面的防御。"

## 2026-07-06: DeepSeek 400 错误——tool_call / tool 消息配对

- **类型**: API 约束 / 技术兼容
- **严重程度**: 阻断
- **现象**: Agent 调用完工具后回到 router，LLM API 报 `400 Bad Request: Messages with role 'tool' must be a response to a preceding message with 'tool_calls'`——整个请求挂掉
- **根因**: DeepSeek（和 OpenAI）的 API 要求：每条 `ToolMessage` 必须在一条带 `tool_calls` 的 `AIMessage` 之后，并且 `tool_call_id` 必须匹配。我们的消息链在多次循环中 tool_call 和 tool 消息的配对被打乱了
- **修复**: `router_node` 每次重新构建 messages 列表——取用户原始问题 + 手动为每个已完成 tool_call 拼上一对 `AIMessage(tool_calls=[...]) → ToolMessage(content=结果, tool_call_id=...)`——不依赖旧消息链
- **面试怎么讲**: "LangGraph 默认用 add reducer 追加消息，但 DeepSeek 对消息顺序和配对有严格要求。我的处理是在 router 节点每次重建消息列表而不是追加——messsage 管理是 Agent 工程中最容易被忽视的坑。"

## 2026-07-06: Streamlit 看不到逐字流式效果

- **类型**: 框架渲染机制 / 体验
- **严重程度**: 体验
- **现象**: SSE 逐字发了（每个 char 一个 event），但页面显示"空白半天 → 突然一大段文字全出来"，没有打字机效果
- **根因**: Streamlit 端用的是 `asyncio.run(service.stream_chat(...))`——直接内存调用，所有 SSE 事件一次性跑完后才触发 Streamlit 的渲染周期。真正走 FastAPI HTTP 的 SSE 端点（`/chat/stream`）是逐字推送的
- **解决方案**: Streamlit 改为 `httpx` HTTP 客户端调 FastAPI，走真正的网络传输通道，SSE 事件逐条到达触发渲染。Phase 4 做
- **面试怎么讲**: "SSE 流式在 HTTP 传输层是逐字推送的，但直接内存调用时会被 asyncio 一次性跑完。解决是让前端走真正的 HTTP 调用后端——这也是微服务架构的优势，前端和后端各自独立。"

## 2026-07-06: 相对路径在不同入口下漂移

- **类型**: 路径解析
- **严重程度**: 阻断
- **现象**: `ModuleNotFoundError: No module named 'src'`。在不同的命令行目录下启动，`./chroma_db` 和 `./bge_model` 找不到，导致看起来"数据库丢失了"
- **根因**: 配置中用了相对路径（`./chroma_db`、`./bge_model`），它们相对于**命令运行时的工作目录**，不是项目根目录。`streamlit run app.py` 和 `uvicorn src.api.server:app` 的工作目录不同，必然有一个找不到
- **修复**: 在 `config.py` 中新增 `_PROJECT_ROOT` 计算（`os.path.dirname(os.path.dirname(os.path.abspath(__file__)))`），所有相对路径统一转为基于项目根目录的绝对路径
- **面试怎么讲**: "配置管理的一个基本原则：不要依赖运行时的工作目录。我用 `__file__` 推算项目根目录，所有路径基于它解析。这样从任何目录启动都不会偏。"

## 2026-07-06: Pydantic 模型定义顺序导致 NameError

- **类型**: Python 语言基本功
- **严重程度**: 阻断
- **现象**: `NameError: name 'DocumentInfo' is not defined`。`DocumentListResponse` 引用了 `DocumentInfo`，但 `DocumentInfo` 定义在它后面
